# IgniteBT Configuration Guide

This guide covers configuring IgniteBT for your development environment, including SDK setup, project configuration, and build system customization. Proper configuration is essential for ensuring that the build system operates correctly and efficiently in your specific development environment. Configuration involves setting up SDKs, defining project structure, specifying build parameters, and tuning performance settings to match your hardware and workflow requirements. This guide provides comprehensive information about all aspects of IgniteBT configuration, from basic setup to advanced customization options.

## Configuration Files

IgniteBT uses several configuration files to manage the build environment, each serving a specific purpose in the overall configuration system. These files work together to provide a comprehensive and flexible configuration framework that can adapt to different project requirements and development environments. Understanding the role of each configuration file and how they interact is important for effective configuration management.

The `.engine` file serves as the project descriptor file, defining the basic structure and fundamental settings of the project. This file is the entry point for the build system, providing information about where to find source code, where to place build outputs, and other essential project metadata. The `IgniteBT.SDKs.json` file contains SDK path configuration, allowing explicit specification of where required SDKs are installed. This file is particularly important when SDKs are installed in non-standard locations or when specific SDK versions need to be used. The `bootstrap.manifest` file contains build system bootstrap configuration, providing the information needed to bootstrap the build system on first use. This file is automatically generated and maintained by the build system, ensuring that the bootstrap process remains consistent with the current build system state.

## Project Configuration

### .engine File

The `.engine` file is the primary project descriptor located in the repository root, defining the project structure and build system settings. This file uses a simple INI-style format with key-value pairs that specify various aspects of the project configuration. The `.engine` file is typically committed to version control as it contains essential project-wide configuration that should be consistent across all development environments. However, certain values may need to be adjusted for local development requirements, and the file format supports both shared and local configuration through appropriate mechanisms.

**Example:**

```ini
# WindEffects engine installation descriptor
schema=1
engine.version=0.1.0
ProgramsRoot=Engine/Source/Programs
BuildRoot=Build
AssetsRoot=Assets
ProjectsRoot=Projects
bootstrap.manifest=Build/Manifest/bootstrap.manifest
bootstrap.entry=IgniteBT
bootstrap.entry.source=IgniteBT/IgniteBT.csproj
```

The configuration parameters in the `.engine` file control fundamental aspects of how the build system operates. The `schema` parameter specifies the descriptor file format version, allowing the build system to handle different versions of the file format and maintain backward compatibility. The `engine.version` parameter provides an engine version identifier that can be used for version-specific behavior or compatibility checks. The `ProgramsRoot` parameter specifies the path to build tools and programs, telling the build system where to find the IgniteBT build system and other development tools. The `BuildRoot` parameter defines the build output directory where all build artifacts will be placed. The `AssetsRoot` parameter specifies the project assets directory where game assets and resources are stored. The `ProjectsRoot` parameter defines the project files directory where project-specific files and configurations are kept. The `bootstrap.manifest` parameter specifies the bootstrap manifest file path, which is used to bootstrap the build system. The `bootstrap.entry` parameter specifies the bootstrap entry point, which is the name of the build system to use. The `bootstrap.entry.source` parameter specifies the bootstrap source project, which is the path to the project file for the build system itself.

**Configuration Parameters:**

- `schema` - Descriptor file format version
- `engine.version` - Engine version identifier
- `ProgramsRoot` - Path to build tools and programs
- `BuildRoot` - Build output directory
- `AssetsRoot` - Project the assets directory
- `ProjectsRoot` - Project files directory
- `bootstrap.manifest` - Bootstrap manifest file path
- `bootstrap.entry` - Bootstrap entry point
- `bootstrap.entry.source` - Bootstrap source project

## SDK Configuration

### IgniteBT.SDKs.json

The SDK configuration file defines paths to required development SDKs, allowing explicit specification of where SDKs are installed on your system. This file is typically located in the repository root and uses JSON format for easy editing and parsing. The SDK configuration file is essential for ensuring that the build system can locate all required dependencies, particularly when SDKs are installed in non-standard locations or when specific SDK versions need to be used. The file supports multiple types of path specifications, including SDK paths for standard development kits, additional include paths for custom libraries, and additional library paths for linking against custom libraries.

**Example:**

```json
{
  "SDKPaths": {
    "VulkanSDK": "F:\\Coding\\windeffects\\Engine\\ThirdParty",
    "SDL3": "F:\\Coding\\windeffects\\Engine\\ThirdParty\\SDL3"
  },
  "AdditionalIncludePaths": {
    "MyCustomLibrary": "C:\\Libraries\\MyCustomLibrary\\include"
  },
  "AdditionalLibraryPaths": {
    "MyCustomLibrary": "C:\\Libraries\\MyCustomLibrary\\lib"
  }
}
```

### Supported SDKs

IgniteBT supports detection and configuration of the following SDKs, which are essential for building the engine and its dependencies. The build system includes automatic detection logic for common SDK installation locations, but also allows explicit configuration through the SDK configuration file. This flexibility ensures that the build system can work with both standard and non-standard SDK installations, accommodating different development environments and organizational requirements.

#### Vulkan SDK

The Vulkan SDK is required for graphics development, as WindEffects Engine uses Vulkan for graphics rendering. IgniteBT automatically detects Vulkan SDK from standard installation paths or uses the configured path when automatic detection is not sufficient. Vulkan SDK detection checks common installation directories on Windows and also respects the VULKAN_SDK environment variable if it is set. Having the Vulkan SDK correctly configured is essential for successful builds, as the engine cannot be built without it.

**Windows Default Paths:**
- `C:\VulkanSDK\[version]`
- Environment variable `VULKAN_SDK`

**Configuration:**

```json
{
  "SDKPaths": {
    "VulkanSDK": "C:\\VulkanSDK\\1.3.290.0"
  }
}
```

#### SDL3

SDL3 (Simple DirectMedia Layer 3) is used for platform abstraction and input handling, providing cross-platform access to windowing, input devices, and other platform-specific services. SDL3 is a critical dependency that enables the engine to work across different platforms with a consistent interface. The build system needs to know where SDL3 is installed to include the appropriate headers and link against the correct libraries.

**Configuration:**

```json
{
  "SDKPaths": {
    "SDL3": "C:\\SDL3"
  }
}
```

### Additional Include and Library Paths

For custom libraries that are not detected automatically by the build system, you can specify additional include and library paths in the SDK configuration file. This capability is important for integrating third-party libraries or custom components that are not part of the standard SDK set. The additional include paths tell the compiler where to find header files for these libraries, while the additional library paths tell the linker where to find the library files for linking. This explicit configuration ensures that custom dependencies can be integrated into the build process without requiring modifications to the build system itself.

```json
{
  "AdditionalIncludePaths": {
    "OpenAL": "C:\\OpenAL\\include",
    "CustomLib": "C:\\Libraries\\CustomLib\\include"
  },
  "AdditionalLibraryPaths": {
    "OpenAL": "C:\\OpenAL\\lib",
    "CustomLib": "C:\\Libraries\\CustomLib\\lib"
  }
}
```

### SDK Detection

IgniteBT automatically detects SDKs from common installation locations, reducing the need for manual configuration. The automatic detection system scans standard installation directories for each supported SDK, checking for version information and validating that the SDK meets the project's requirements. This automatic detection makes it easier to get started with the build system, as most SDKs will be found without manual configuration. However, when automatic detection fails or when you need to use a specific SDK installation, you can trigger detection manually or configure explicit paths.

To trigger detection manually, use the sdk detect command, which forces the build system to rescan for SDKs and update its internal database of available SDKs. This is useful when you have installed or updated SDKs after the initial setup.

```powershell
we sdk detect
```

To validate SDK configuration, use the sdk validate command, which checks that all required SDKs are present and meet the project's version requirements. This validation helps catch configuration issues before they cause build failures.

```powershell
we sdk validate
```

To list all detected SDKs, use the sdk list command, which shows which SDKs have been found, their versions, and their installation paths. This provides visibility into what the build system has detected and can help identify missing or misconfigured SDKs.

```powershell
we sdk list
```

## Build Configuration

### Build Configurations

IgniteBT supports three standard build configurations, each optimized for different scenarios and stages of the development lifecycle. These configurations control compiler optimization levels, debug information inclusion, and other build characteristics that affect both build time and runtime behavior. Choosing the appropriate configuration for your current task is important for optimizing your development workflow and ensuring that you have the right balance of performance, debugging capability, and build time.

#### Debug

The Debug configuration provides maximum debugging information with no compiler optimizations, making it ideal for debugging complex issues that require detailed inspection of program state. This configuration includes comprehensive debugging symbols that allow you to inspect variables, step through code line by line, and analyze program execution in detail. Compiler optimizations are completely disabled to ensure that the generated code matches the source code as closely as possible, which is crucial for accurate debugging when you need to understand exactly what the code is doing. However, this comes at the cost of slow execution speed and large binary size, as the lack of optimizations and the inclusion of extensive debugging information significantly impact performance and file size. The Debug configuration is typically used only when actively debugging difficult issues that cannot be resolved with the Development configuration.

- Maximum debugging information
- No optimizations
- Slowest execution
- Largest binary size

**Use for:** Complex debugging sessions

#### Development

The Development configuration provides an optimized build with debugging symbols, making it the recommended choice for daily development and testing. This configuration strikes an optimal balance between performance and debuggability, applying compiler optimizations to improve runtime performance while still including debugging symbols for debugging when needed. The inclusion of debugging symbols means that you can still attach a debugger and inspect program state, but the optimizations mean that the code runs faster and the binaries are smaller than in the Debug configuration. This configuration provides good performance for testing and iteration while still maintaining reasonable debugging capabilities. The Development configuration is the default for most development work, as it provides the best compromise between build time, runtime performance, and debugging capability.

- Debugging symbols included
- Compiler optimizations enabled
- Good performance
- Reasonable binary size

**Use for:** Daily development and testing

#### Shipping

The Shipping configuration provides a fully optimized release build with no debugging information, making it suitable for final releases and distribution. This configuration excludes debugging information entirely and applies maximum compiler optimizations to produce the smallest and fastest possible binaries. The absence of debugging information reduces binary size and eliminates the possibility of debugging the release build, but this is appropriate for shipped software where debugging is not needed. The maximum optimizations provide the best possible runtime performance, which is important for user experience. The Shipping configuration should be used for final builds that will be distributed to users, as it provides the optimal combination of small size and fast execution.

- No debugging information
- Maximum optimizations
- Best performance
- Smallest binary size

**Use for:** Release builds and distribution

### Platform Configuration

IgniteBT supports multiple target platforms, allowing the engine to be built for different operating systems and architectures. Cross-platform support is essential for projects that need to target multiple platforms or that have team members working on different operating systems. The build system abstracts platform-specific differences, allowing developers to use a consistent set of commands across all supported platforms while still producing platform-appropriate binaries.

- **Win64** - Windows 64-bit (default)
- **Windows** - Windows (platform-agnostic)
- **Linux** - Linux 64-bit
- **Mac** - macOS 64-bit

Specify platform with the platform flag when invoking build commands:

```powershell
we build --config Development --platform Win64
```

### Compiler Configuration

IgniteBT uses platform-appropriate compilers for each target platform, automatically selecting the correct compiler and configuring it appropriately for the platform. This abstraction allows developers to use the same build commands across different platforms without needing to know the specific compiler details for each platform. The build system handles the differences in compiler command-line syntax, flag names, and output formats, presenting a unified compiler model to the rest of the system.

- **Windows** - MSVC (Microsoft Visual C++)
- **Linux** - Clang
- **macOS** - Clang

Compiler-specific settings are managed automatically based on the target platform, ensuring that the correct compiler flags, preprocessor definitions, and other compiler-specific settings are applied without manual configuration.

## Module Configuration

### Module Rules

Build modules are configured through module rule files, which define compilation settings, dependencies, and output configuration for each module. Module rules are the primary mechanism for controlling how individual components of the project are built, allowing fine-grained control over compilation behavior at the module level. These rules are discovered automatically from the source tree, meaning that the build system scans the project structure to find module rule files and registers the modules they define without requiring manual registration.

Module rules define several important aspects of how a module is built. Source file inclusion patterns specify which source files should be included in the module, allowing selective inclusion based on file patterns or directories. Compiler flags and defines specify the compiler flags and preprocessor definitions that should be used when compiling the module, allowing module-specific compilation settings. Public and private dependencies specify which other modules this module depends on, distinguishing between dependencies that are exposed to consumers of the module and dependencies that are internal to the module. Precompiled header configuration specifies whether and how precompiled headers should be used for the module, which can significantly improve compilation speed. Unity build settings specify whether unity builds should be used for the module and how source files should be batched together.

Module rules are discovered automatically from the source tree and define:

- Source file inclusion patterns
- Compiler flags and defines
- Public and private dependencies
- Precompiled header configuration
- Unity build settings

### Module Dependencies

Modules declare dependencies on other modules to specify the relationships between different components of the project. These dependencies are essential for ensuring that modules are built in the correct order and that the necessary linking information is available when producing final binaries. IgniteBT resolves these dependencies automatically, ensuring correct build order and preventing issues such as undefined symbols or missing libraries.

Dependencies can be classified as public or private. Public dependencies are dependencies that are exposed to consumers of the module, meaning that any module that depends on this module also transitively depends on the public dependencies. This is appropriate for dependencies that are part of the module's public API or that are required to use the module. Private dependencies are dependencies that are internal to the module and are not exposed to consumers. These are used for implementation details that should not affect the module's public interface.

**Example dependency declaration:**

```csharp
PublicDependencies = new[] { "WECore", "WERenderer" };
PrivateDependencies = new[] { "ThirdParty/SDL3" };
```

### Module Output Configuration

Modules specify their output type, which determines what kind of build artifact is produced when the module is built. The output type is a fundamental characteristic of a module that affects how it is linked and how it can be used by other modules.

- **Executable** - Standalone applications such as the Editor or Game executable. Executable modules produce programs that can be run directly and are typically the entry points for the application.
- **Shared Library** - DLLs on Windows or shared objects on Linux/macOS. Shared libraries are loaded at runtime and can be used by multiple executables simultaneously.
- **Static Library** - LIB files on Windows or archive files on Linux/macOS. Static libraries are linked directly into executables or other libraries at link time.

## Environment Configuration

### Environment Variables

Configure IgniteBT behavior with environment variables, providing a flexible way to set default behavior without needing to specify options on every command. Environment variables are particularly useful for setting up consistent defaults across your development environment or for configuring CI/CD pipelines where command-line options may be inconvenient to manage. Environment variables can be set at the system level, user level, or session level, allowing different scopes of configuration depending on your needs.

#### Build Configuration

Build configuration environment variables allow you to set defaults for commonly used build parameters, reducing the need to specify these parameters repeatedly. The `IGNITEBT_CONFIG` environment variable sets the default build configuration, allowing you to specify a preferred configuration such as Development or Shipping that will be used unless explicitly overridden on the command line. This is useful for establishing a standard configuration for your daily development work. The `IGNITEBT_PLATFORM` environment variable sets the default target platform, allowing you to specify a preferred platform such as Win64 or Linux. The `IGNITEBT_JOBS` environment variable sets the default parallel job count, allowing you to specify a preferred level of parallelism that will be used for builds unless overridden.

```powershell
# Default build configuration
set IGNITEBT_CONFIG=Development

# Default target platform
set IGNITEBT_PLATFORM=Win64

# Default parallel job count
set IGNITEBT_JOBS=8
```

#### Output Configuration

Output configuration environment variables control where build artifacts are placed and how build output is displayed. The `IGNITEBT_BUILD_ROOT` environment variable specifies a custom build directory path, allowing you to place build artifacts in a location other than the default. This is useful for separating build artifacts from source code or for placing build artifacts on a different drive for performance reasons. The `IGNITEBT_VERBOSE` environment variable enables verbose output globally when set to `1`, providing detailed output for all commands without needing to specify the `--verbose` flag on each command. This is particularly useful for troubleshooting or when you want detailed information about all build operations.

```powershell
# Custom build directory
set IGNITEBT_BUILD_ROOT=C:\Builds\WindEffects

# Enable verbose output globally
set IGNITEBT_VERBOSE=1
```

#### SDK Configuration

SDK configuration environment variables provide an alternative mechanism for specifying SDK paths, which can be useful for overriding automatic detection or for configuring SDKs in environments where modifying configuration files is not practical. The `VULKAN_SDK` environment variable allows you to override the Vulkan SDK path, specifying where the Vulkan SDK is installed. This environment variable is checked by the build system in addition to the paths specified in the SDK configuration file, providing a flexible override mechanism.

```powershell
# Override Vulkan SDK path
set VULKAN_SDK=C:\VulkanSDK\1.3.290.0
```

### Persistent Configuration

For persistent configuration across sessions, environment variables can be set in various locations depending on your operating system and preferences. On Windows, you can set environment variables through the System Properties interface, which provides a graphical interface for managing system and user environment variables. This approach makes the variables persistent across all sessions and applications. For PowerShell users, environment variables can be added to the PowerShell profile script, which is executed every time a PowerShell session starts. This allows for session-specific configuration that applies only to PowerShell. For projects that support it, a `.env` file can be used to define environment variables that are automatically loaded by development tools and IDEs. This approach keeps configuration project-specific and version-controlled.

- **Windows**: System Properties → Environment Variables
- **PowerShell Profile**: `$PROFILE` script
- **Project**: `.env` file (if supported by your tooling)

## Build Layout Configuration

### Directory Structure

The build output directory structure is configurable through the build layout system, providing a flexible framework for organizing build artifacts. The default structure is designed to be logical and efficient, separating different types of artifacts to avoid conflicts and make it easy to locate specific outputs. However, the layout system can be customized to match specific project requirements or organizational standards.

```
Build/
├── Output/              # Final binaries
├── Intermediate/        # Object files
├── Generated/           # Generated source
├── Cache/              # Build cache
├── Database/           # Metadata
├── Logs/               # Build logs
└── Manifest/           # Build manifests
```

The Output directory contains the final binaries and libraries that are the primary products of the build process. These are the files that developers and users actually interact with, such as executable programs, shared libraries, and static libraries. The Intermediate directory contains object files and compilation artifacts that are intermediate products of the compilation process. These files are not typically used directly but are essential for incremental builds. The Generated directory contains source files that are generated during the build process, such as export definition files and generated code. The Cache directory contains the build cache for incremental builds, storing compiled artifacts and dependency information to speed up subsequent builds. The Database directory contains build metadata and dependency information, including the dependency graph and build history. The Logs directory contains build and runtime logs, providing a record of build operations for troubleshooting and analysis. The Manifest directory contains build manifests and descriptors, which describe the build process and its outputs.

### Custom Build Root

To use a custom build directory, you can set the environment variable to specify an alternative location for build artifacts. This is useful for separating build artifacts from source code, placing build artifacts on a different drive for performance reasons, or maintaining separate build directories for different configurations or branches.

```powershell
set IGNITEBT_BUILD_ROOT=D:\Builds\WindEffects
```

Alternatively, you can modify the `.engine` file to specify a custom build root directly in the project configuration. This approach makes the custom build root part of the project configuration, which can be useful for ensuring consistency across different development environments.

```ini
BuildRoot=D:\Builds\WindEffects
```

## Distributed Build Configuration

### Worker Configuration

For distributed builds, configure worker processes to handle build tasks across multiple machines. The daemon command starts the IgniteBT daemon for background build operations and distributed compilation coordination. The daemon is a background process that provides build services without requiring manual initiation of each build operation. This is particularly useful for distributed builds, where the daemon coordinates work across multiple machines, and for continuous integration scenarios where builds need to run automatically. The daemon listens for build requests, manages worker processes, and handles the complexity of distributed build coordination.

```powershell
we daemon --workers 4 --port 8080
```

The workers parameter specifies the number of worker processes to spawn, controlling the level of parallelism available for background builds. More workers can process more builds concurrently but consume more system resources. The port parameter specifies the port number on which the daemon should listen for connections, allowing customization to avoid port conflicts or to integrate with existing infrastructure.

### Cache Configuration

Distributed build cache settings enable sharing of build artifacts across multiple build machines, reducing redundant work and improving overall build efficiency. The distributed cache is a key component of the distributed build system, allowing build artifacts to be shared across multiple build machines. When a worker completes a task, the resulting artifact is stored in the distributed cache, making it available to other workers that may need the same artifact. This sharing is particularly valuable for common dependencies that are used by many different parts of the project.

```json
{
  "DistributedCache": {
    "Enabled": true,
    "ServerAddress": "build-cache.example.com",
    "Port": 9090,
    "Authentication": "token"
  }
}
```

The Enabled flag controls whether the distributed cache is active. When disabled, each build machine operates independently with its own local cache. The ServerAddress specifies the hostname or IP address of the distributed cache server. The Port specifies the port number on which the cache server is listening. The Authentication parameter specifies the authentication method and credentials to use when connecting to the cache server, ensuring that only authorized build machines can access the cache.

## Performance Tuning

### Parallel Job Configuration

Optimizing the parallel job count based on your system's capabilities is essential for achieving optimal build performance. The optimal number of parallel jobs depends on factors such as the number of CPU cores, available memory, and the nature of the build. Using more parallel jobs can reduce build time by utilizing more CPU cores, but using too many can actually slow down the build due to resource contention and overhead. For CPU-bound builds, where compilation is the primary bottleneck, using a job count equal to or slightly higher than the number of CPU cores typically provides the best performance. For I/O-bound builds, where disk I/O is the primary bottleneck, using fewer parallel jobs can reduce I/O contention and improve performance.

```powershell
# For CPU-bound builds
we build --jobs 8

# For I/O-bound builds
we build --jobs 4

# Use all available cores
we build --jobs 0
```

When you specify a job count of zero, IgniteBT automatically determines the optimal number of jobs based on your system's hardware capabilities. This automatic detection typically uses the number of logical CPU cores as a starting point and may adjust based on available memory and other factors.

### Cache Configuration

Build cache settings affect incremental build performance by controlling how build Artifacts are stored and reused. The cache is a critical component of the build system's performance, as it allows the system to avoid redundant work by reusing artifacts from previous builds. Configuring the cache appropriately can significantly improve build times, especially for incremental builds where only small changes are made between builds.

```json
{
  "Cache": {
    "Enabled": true,
    "MaxSizeGB": 50,
    "CompressionEnabled": true
  }
}
```

The Enabled flag controls whether the build cache is active. When disabled, the build system cannot reuse artifacts from previous builds, resulting in slower incremental builds. The MaxSizeGB parameter specifies the maximum size of the cache in gigabytes, allowing you to control how much disk space the cache can consume. This is important for preventing the cache from growing too large and consuming excessive disk space. The CompressionEnabled flag controls whether cache entries are compressed to reduce disk usage. Compression can significantly reduce cache size but may add CPU overhead during cache operations.

### Precompiled Headers

Enable precompiled headers for faster compilation by reducing the time spent parsing header files. Precompiled headers are a powerful optimization for C++ projects, where header files can be large and complex, and the same headers are often included in many source files. By precompiling commonly used headers, the build system can avoid re-parsing these headers for every source file, dramatically reducing compilation time.

```csharp
PCHEnabled = true;
PCHHeader = "PCH.h";
PCHSource = "PCH.cpp";
```

The PCHEnabled flag enables precompiled headers for the module. The PCHHeader specifies the header file that should be precompiled, typically a header that includes all the commonly used headers for the module. The PCHSource specifies the source file that is used to generate the precompiled header, which must include the PCHHeader as its first include.

### Unity Builds

Configure unity builds to reduce compilation overhead by combining multiple source files into fewer compilation units. Unity builds are an optimization technique that reduces the overhead of parsing headers by compiling multiple source files together as a single translation unit. This reduces the number of times headers need to be parsed and can significantly improve compilation speed, especially for projects with many small source files.

```csharp
UnityBuildEnabled = true;
UnityBuildBatchSize = 16;
```

The UnityBuildEnabled flag enables unity builds for the module. The UnityBuildBatchSize specifies how many source files should be combined into each unity build batch. Larger batch sizes can reduce header parsing overhead but may increase memory usage during compilation.

## Diagnostic Configuration

### Logging Configuration

Configure logging verbosity and output to control the amount of information displayed during build operations. Logging is an essential diagnostic tool that provides visibility into the build process, helping identify issues and understand what the build system is doing. The level of logging detail can be adjusted based on your needs, from minimal output for normal operations to verbose output for troubleshooting.

```powershell
# Enable verbose logging
we build --verbose

# Set log level via environment
set IGNITEBT_LOG_LEVEL=Verbose
```

Verbose logging provides detailed information about the build process, including the exact compiler commands being executed, file-by-file compilation progress, dependency resolution details, and linker output. This level of detail is invaluable for troubleshooting build failures or when trying to understand the impact of changes on the build process. The IGNITEBT_LOG_LEVEL environment variable provides a way to set the default logging level globally, which is useful for ensuring consistent logging behavior across all build operations.

### Build Reports

Generate detailed build reports to analyze build performance and identify optimization opportunities. Build reports provide comprehensive information about the build process, including timing information for each phase, dependency analysis, and resource utilization. These reports are valuable for understanding where time is being spent during builds and identifying bottlenecks that can be addressed to improve performance.

```powershell
we build --report
```

Reports are saved to `Build/Logs/` with detailed timing and dependency information. The reports include information such as total build time, time spent in each build phase, compilation times for individual files, linking times, and cache hit rates. Analyzing these reports can help identify slow modules, inefficient dependency chains, and opportunities for performance optimization through configuration changes.

## Validation and Testing

### Configuration Validation

Validating your configuration is an important step to ensure that the build system is correctly configured and that all required dependencies are available. Regular validation helps catch configuration issues before they cause build failures, saving time and frustration. The validation process checks various aspects of the configuration, including SDK availability, tool installation, file system permissions, and dependency integrity.

```powershell
# Run full diagnostics
we doctor

# Validate SDK configuration
we sdk validate

# Validate project configuration
we project validate
```

The doctor command performs a comprehensive check of the build environment, verifying that all required tools and SDKs are installed and properly configured. This proactive check helps identify issues before they cause build failures. The sdk validate command checks that all required SDKs are present and meet the project's version requirements, catching version incompatibilities before they cause build failures. The project validate command checks the project configuration for consistency and correctness, ensuring that the project structure is properly defined and that all required configuration files are present and valid.

### Test Builds

Testing your configuration with a clean build is the ultimate validation step, as it verifies that the configuration actually works in practice. A clean build eliminates any potential issues from stale or corrupted build artifacts, ensuring that the build can complete successfully from a fresh start. This is particularly important after making configuration changes or when setting up a new development environment.

```powershell
we clean
we build --config Development --verbose
```

The clean command removes all build artifacts, forcing a complete rebuild on the next build. This eliminates any potential issues from stale artifacts. The build command with verbose output provides detailed information about the build process, making it easier to identify any issues that occur during the build. Running a clean build with verbose output is the most thorough way to validate that your configuration is working correctly.

## Common Configuration Scenarios

### Development Workstation

The development workstation configuration is optimized for daily development work, prioritizing iteration speed and debugging capability over absolute performance. This configuration uses the Development configuration, which provides a good balance between optimization and debugging capability. The parallel job count is set to a reasonable number that provides good parallelism without causing excessive resource contention. Verbose output is disabled by default to keep build output concise, but it can be enabled when needed for troubleshooting.

```powershell
set IGNITEBT_CONFIG=Development
set IGNITEBT_JOBS=8
set IGNITEBT_VERBOSE=0
```

This configuration is ideal for most development work, providing fast incremental builds while still maintaining the ability to debug issues when they arise. The Development configuration includes debugging symbols and applies optimizations, making it suitable for both development and testing.

### CI/CD Pipeline

The CI/CD pipeline configuration is optimized for automated builds, prioritizing reproducibility, reliability, and detailed logging over iteration speed. This configuration uses the Shipping configuration to produce release-quality builds. The parallel job count is set to zero to allow the build system to automatically determine the optimal level of parallelism based on the available hardware, which is important in CI environments where hardware characteristics may vary. Verbose output is enabled to provide detailed logs for troubleshooting if issues occur during the automated build process.

```powershell
set IGNITEBT_CONFIG=Shipping
set IGNITEBT_JOBS=0
set IGNITEBT_VERBOSE=1
```

This configuration ensures that CI builds are consistent and reproducible, with detailed logs available for analysis if builds fail. The Shipping configuration produces optimized binaries suitable for distribution, and the verbose logging provides the information needed to diagnose any issues that occur during the automated build process.

### Release Build

The release build configuration is optimized for final releases, prioritizing performance and binary size over debugging capability. This configuration uses the Shipping configuration to produce fully optimized release builds. The parallel job count is set to a high value to maximize build speed for release builds, which are typically run less frequently than development builds. A custom build root is used to separate release builds from development builds, preventing conflicts and making it easier to manage release artifacts.

```powershell
set IGNITEBT_CONFIG=Shipping
set IGNITEBT_JOBS=16
set IGNITEBT_BUILD_ROOT=D:\Releases\WindEffects
```

This configuration produces the highest quality release binaries with maximum optimizations and no debugging information. The custom build root ensures that release artifacts are kept separate from development artifacts, making it easier to manage the release process and preventing accidental contamination of release builds with development artifacts.

## Troubleshooting Configuration

### SDK Not Found

If an SDK is not detected by the build system, this typically indicates that the SDK is not installed in a standard location or that the SDK configuration is incorrect. SDK detection issues are a common cause of build failures, particularly for graphics-related builds that depend on specific SDKs like Vulkan. Resolving SDK detection issues involves verifying the SDK installation, checking the configuration, and refreshing the detection process.

1. Verify SDK installation by confirming that the SDK is actually installed on your system and checking the installation directory. Make sure that the SDK version meets the project's requirements.
2. Check `IgniteBT.SDKs.json` paths to ensure that the paths specified in the configuration file are correct and point to valid SDK installations. Paths should use proper escaping for backslashes in JSON format.
3. Run `we sdk detect` to refresh the detection process, forcing the build system to rescan for SDKs and update its internal database. This is useful when SDKs have been installed or updated after the initial setup.
4. Run `we sdk validate` to verify that the SDK configuration the correct and that all required SDKs can now be located. This validation step confirms that the configuration changes have resolved the detection issue.

### Build Path Issues

If build paths are incorrect, this can cause the build system to place artifacts in unexpected locations or fail to find required files. Build path issues can arise from incorrect configuration in the `.engine` file, environment variable overrides, or file system permission problems. Resolving build path issues involves checking the configuration, verifying environment variables, and ensuring that the build directory is accessible.

1. Check `BuildRoot` in `.engine` file to ensure that the build root path is correctly specified and points to a valid location. The path should be relative to the repository root or an absolute path that exists on your system.
2. Verify `IGNITEBT_BUILD_ROOT` environment variable to ensure that it is not overriding the build root specified in the `.engine` file. If the environment variable is set, it takes precedence over the configuration file.
3. Ensure write permissions to build directory by verifying that your user account has permission to create and modify files in the build directory. Build operations require write access to create intermediate files, output files, and cache data.
4. Check for path length limitations on Windows, as Windows has a maximum path length limitation that can cause issues with deep directory structures. If paths are too long, consider using a shorter build root path closer to the drive root.

### Module Not Found

If a module is not discovered by the build system, this typically indicates that the module is not in the expected location or that the module rule file has syntax errors. Module discovery is automatic based on the source tree structure, so issues with module discovery usually indicate problems with the module configuration or the project structure.

1. Verify module is in source tree by confirming that the module directory and its files are located in the expected location under the ProgramsRoot specified in the `.engine` file. Modules must be in the correct location to be discovered automatically.
2. Check module rule file syntax to ensure that the module rule file is syntactically correct and follows the expected format. Syntax errors can prevent the module from being parsed and registered with the build system.
3. Run `we modules` to list discovered modules, which will show which modules the build system has successfully discovered. This can help identify whether the module is missing entirely or if there is an issue with a specific module.
4. Check `ProgramsRoot` configuration to ensure that it points to the correct directory where modules are located. If the ProgramsRoot is incorrect, the build system will not be able to find modules even if they exist in the repository.

## Advanced Configuration

### Custom Build Steps

Add custom build steps through module rules to extend the build process with project-specific operations. Custom build steps allow you to execute arbitrary commands as part of the build process, which is useful for code generation, asset processing, or other operations that need to be integrated into the build. Custom build steps are executed at specific points in the build process, allowing you to hook into the build workflow at appropriate times.

```csharp
CustomBuildSteps = new[]
{
    new CustomBuildStep
    {
        Command = "python",
        Arguments = "generate_code.py",
        WorkingDirectory = "$(ModuleDir)"
    }
};
```

The Command parameter specifies the executable to run for the custom build step. This can be any executable available on the system, including scripts, compilers, or custom tools. The Arguments parameter specifies the command-line arguments to pass to the executable. The WorkingDirectory parameter specifies the directory in which the command should be executed, with $(ModuleDir) being a variable that expands to the module's directory. Custom build steps are executed in the order they are defined, and their output is logged along with the rest of the build output.

### Conditional Compilation

Configure conditional compilation based on platform or configuration to enable or disable code based on build parameters. Conditional compilation is a powerful technique for creating code that can be built for different platforms or configurations without maintaining separate codebases. By defining preprocessor symbols based on build parameters, you can include or exclude code sections at compile time.

```csharp
Defines = new[]
{
    "PLATFORM_WINDOWS",
    "CONFIG_DEVELOPMENT",
    "ENABLE_PROFILING"
};
```

The Defines parameter specifies a list of preprocessor definitions that should be passed to the compiler. These definitions can be used in source code with conditional compilation directives to include or exclude code based on the build configuration. Platform-specific definitions like PLATFORM_WINDOWS allow you to write platform-specific code that is only compiled for the target platform. Configuration-specific definitions like CONFIG_DEVELOPMENT allow you to include debugging or development code only in specific configurations. Feature definitions like ENABLE_PROFILING allow you to optionally include features based on configuration.

### Toolchain Customization

Customize compiler and linker flags to fine-tune the build process for specific requirements. While the build system provides sensible defaults for compiler and linker flags, there may be situations where you need to customize these flags for specific modules or for the entire project. Toolchain customization allows you to add additional flags or override the default flags to meet specific requirements.

```csharp
CompilerFlags = new[] { "/W4", "/WX" };
LinkerFlags = new[] { "/INCREMENTAL:NO" };
```

The CompilerFlags parameter specifies additional compiler flags that should be passed to the compiler. These flags are added to the default flags provided by the build system. In the example, /W4 sets the warning level to 4, which enables high warning levels, and /WX treats warnings as errors, which forces the build to fail if any warnings are generated. The LinkerFlags parameter specifies additional linker flags that should be passed to the linker. These flags are added to the default linker flags. In the example, /INCREMENTAL:NO disables incremental linking, which can be useful for release builds where incremental linking is not desired.

## Configuration Best Practices

Following configuration best practices ensures that your build system configuration is maintainable, reliable, and works correctly across different development environments. These practices are based on experience with real-world build systems and address common pitfalls that can lead to configuration issues, build failures, or difficulties for other developers setting up the environment.

1. **Version Control Configuration Files** - Commit `.engine` and `IgniteBT.SDKs.json` (with local paths excluded) to version control to ensure that all developers use a consistent configuration. Version controlling configuration files makes it easier to maintain consistency across the team and provides a history of configuration changes. However, local paths that are specific to individual machines should be excluded from version control, typically through .gitignore patterns or by using environment variables for machine-specific settings.

2. **Use Relative Paths** - Prefer relative paths in configuration files rather than absolute paths. Relative paths are more portable across different development environments and make it easier to move the repository to different locations without breaking the configuration. Relative paths also make it easier for other developers to use the configuration without needing to adjust paths for their local environment.

3. **Document Custom Settings** - Add comments explaining non-standard configurations to help other developers understand why specific configuration choices were made. Documentation is particularly important for unusual or non-obvious configuration settings that may not be immediately understandable to someone unfamiliar with the project. Comments should explain the rationale for the setting and any dependencies or implications.

4. **Validate Regularly** - Run `we doctor` periodically to catch configuration drift and ensure that the configuration remains valid as the project evolves. Configuration drift can occur when SDKs are updated, tools are changed, or the project structure is modified. Regular validation helps catch these issues early before they cause build failures.

5. **Separate Local Settings** - Use environment variables for machine-specific settings rather than hardcoding them in configuration files. This allows each developer to customize their environment without modifying shared configuration files, reducing merge conflicts and making it easier to maintain consistent shared configuration.

6. **Test Clean Builds** - Periodically test clean builds to ensure configuration completeness and that the build can succeed from a fresh start. Clean builds eliminate the influence of cached artifacts and can reveal configuration issues that are masked during incremental builds. Testing clean builds on a regular basis ensures that the configuration is complete and that new developers can successfully build the project.

## Security Considerations

Security is an important consideration when configuring build systems, as build systems often handle sensitive information, execute with elevated privileges, and interact with external systems. Following security best practices helps protect your development environment, prevent unauthorized access, and ensure that build artifacts are not compromised.

- **Sensitive Paths** - Avoid committing paths containing sensitive information to version control. This includes paths that contain credentials, API keys, or other sensitive data. Sensitive information should be stored in environment variables or in configuration files that are excluded from version control. This prevents sensitive information from being exposed through version control history or to unauthorized users who have access to the repository.

- **Access Control** - Ensure build directories have appropriate permissions to prevent unauthorized access to build artifacts. Build directories may contain sensitive information such as debug symbols or intermediate files that could provide information about the codebase. Setting appropriate file permissions ensures that only authorized users can access these files.

- **SDK Validation** - Validate SDK integrity before use to ensure that SDKs have not been tampered with or corrupted. SDKs should be obtained from trusted sources and verified using checksums or digital signatures when available. This prevents the use of compromised SDKs that could introduce security vulnerabilities into the build.

- **Network Configuration** - Secure distributed build cache connections to prevent unauthorized access to build artifacts and to protect the integrity of the cache. Distributed build caches should use authentication and encryption to ensure that only authorized build machines can access the cache and that cache contents cannot be intercepted or modified in transit.

## Migration Guide

### Upgrading Configuration

When upgrading IgniteBT to a new version, it is important to carefully manage the configuration migration process to ensure that existing configurations continue to work correctly and to take advantage of new features. Upgrades may introduce breaking changes, deprecated settings, or new configuration options that need to be understood and applied. Following a systematic upgrade process helps minimize disruption and ensures a smooth transition to the new version.

1. Backup existing configuration files before making any changes. This is the most important step, as it provides a safety net in case the upgrade introduces issues that need to be rolled back. Back up the `.engine` file, `IgniteBT.SDKs.json`, and any other configuration files that you have customized. Store these backups in a safe location where they can be easily restored if needed.

2. Review changelog for breaking changes to understand what has changed in the new version. The changelog will document any breaking changes, deprecated features, and new configuration options. Pay particular attention to breaking changes that may require updates to your configuration. Understanding these changes before starting the upgrade process helps you prepare for any necessary modifications.

3. Run `we doctor` to validate new configuration after applying any required changes. The doctor command will check that the configuration is valid with the new version of IgniteBT and will identify any issues that need to be addressed. This validation step is crucial for catching configuration problems before they cause build failures.

4. Test clean build with new configuration to ensure that everything works correctly. A clean build eliminates the influence of cached artifacts and tests the full build process from start to finish. This is the most comprehensive way to validate that the upgrade was successful and that the configuration is working correctly with the new version.

5. Update any deprecated settings to use the new recommended approaches. Deprecated settings may still work in the current version but will be removed in a future version, so it is important to update them promptly. The changelog or documentation will provide guidance on what deprecated settings should be replaced with and how to make the transition.

### Legacy Configuration

For projects with legacy build systems, IgniteBT provides migration tools to import existing configurations, making it easier to transition from older build systems to IgniteBT. Migration tools can import configuration from various sources, including other build systems, project files, and custom configuration formats. The migration process analyzes the existing configuration and generates equivalent IgniteBT configuration files, preserving as much of the original configuration as possible while adapting it to IgniteBT's configuration model.

Migration tools handle common configuration elements such as module definitions, dependencies, compiler flags, and build settings. However, complex or custom configurations may require manual adjustment after the automated migration. It is important to review the migrated configuration carefully to ensure that it accurately represents the original configuration and to make any necessary adjustments. Testing the migrated configuration with a clean build is essential to validate that the migration was successful and that the build works correctly with the new configuration.
